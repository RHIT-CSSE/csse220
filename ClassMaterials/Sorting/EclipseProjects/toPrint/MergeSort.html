  139 // Question on runtime:
  140 // What is the best and worst case running time of this algorithm? Provide
  141 // formulae for the number of times compareTo() gets called and perform the
  142 // big-Oh analysis of the running times.
  143 //
  144 // compareTo() is only called when two sorted halves are being merged.
  145 // The worst case occurs when the largest two elements are each in a different
  146 // half of the array. This is (n-1) times.
  147 // When merging to obtain two arrays of size n/2, it takes 2*(n/2 - 1) or n-2
  148 // comparisons.
  149 // Merging 4 arrays of size n/4 takes 4*(n/4 - 1) = n - 4 comparisons.
  150 // Merging 2^i arrays of size n / 2^i takes n - 2^i comparisons.
  151 // This continues for m = log(n) rounds, since the number of times that you can
  152 // divide n by 2 until you reach 1 is the solution to n/(2^m) = 1, or m =
  153 // log_2(n).
  154 // So the sum (i = 1, i < m, n - 2^i) ~ nlogn - n = O(nlogn)
  155 //
  156 // The best case is when all the array elements in one half are less than all
  157 // the elements in the other half. Then it only executes n/2 comparisons.
  158  
  159 // Merging to find two arrays of size n/2 takes 2*(n/4) = n/2.
  160 // Merging 4 arrays of size n/4 takes 4*(n/8) = n/2.
  161 // The depth of the recursion is still log n, so the total runtime is
